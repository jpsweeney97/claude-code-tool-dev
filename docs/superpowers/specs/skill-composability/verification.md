### Interim Materiality Verification Protocol

Until the materiality harness is implemented, any implementation of materiality evaluation logic MUST be manually walked through:

1. The 24-case validity matrix (3 `affected_surface` values Ã— 2 `material` values Ã— 4 `suggested_arc` values). For each case: verify the tuple either passes through uncorrected (valid) or is corrected per rules 1-5 (invalid).
2. The 4 consequence prohibitions with explicit fixture inputs:
   - Input: `affected_surface=diagnosis`, `material=true`, `suggested_arc=next-steps` â†’ expected: corrected to `adversarial-review` (rule 3). Prohibition: `diagnosis` MUST NOT emit `next-steps`.
   - Input: `affected_surface=planning`, `material=true`, `suggested_arc=adversarial-review` â†’ expected: corrected to `next-steps` (rule 4). Prohibition: `planning` MUST NOT emit `adversarial-review`.
   - Input: `affected_surface=evidence-only`, `material=true`, `suggested_arc=adversarial-review` â†’ expected: corrected to `dialogue_continue` (rule 2). Prohibition: `evidence-only` MUST NOT emit AR, NS, or `ambiguous`.
   - Input: `affected_surface=diagnosis`, `material=true`, `suggested_arc=dialogue_continue` â†’ expected: corrected to `adversarial-review` (rule 3 applies; rule 1 does not apply since material=true; this exercises prohibition 4: material diagnosis/planning MUST NOT silently remain `dialogue_continue`). Verify rule 3 fires.
   - Input: `affected_surface=planning`, `material=true`, `suggested_arc=dialogue_continue` â†’ expected: corrected to `next-steps` (rule 4 applies; rule 1 does not apply since material=true; this exercises prohibition 4 for the `planning` surface). Verify rule 4 fires.
3. The novelty veto: one case where a novel item pattern-matches a Tier 1 exclusion class â†’ verify Tier 1 returns `no_match`.
4. The cross-tier guard: one case where Tier 1 model exclusion is overridden by Tier 2 inclusion â†’ verify `material: true`.
5. The Step 0 precondition: verify that materiality evaluation checks the `upstream_handoff` pipeline state. Four cases (with dedicated post-abort assertions for case c): (a) `upstream_handoff` present with `tautology_filter_applied: true` â†’ materiality evaluation proceeds, (b) no `upstream_handoff` â†’ precondition trivially satisfied, materiality evaluation proceeds, (c) `upstream_handoff` present with `decomposition_seed` used AND `tautology_filter_applied: false` â†’ materiality evaluation aborted with structured warning and capsule assembly aborted â€” verify: (i) no `<!-- dialogue-feedback-capsule:v1 -->` sentinel appears in output, (ii) no capsule body is emitted, (iii) no durable file written at `.claude/composition/feedback/`, (iv) a structured warning containing "tautology_filter_applied" or "precondition" appears in prose output, (d) `upstream_handoff` present with `decomposition_seed` not used (`--plan` not set) â†’ precondition satisfied, materiality evaluation proceeds (tautology filter not applicable to non-decomposition scenarios).
   Case (c) dedicated post-abort verification (all 5 assertions): (i) no `<!-- dialogue-feedback-capsule:v1 -->` sentinel appears in output, (ii) no capsule body is emitted, (iii) no durable file written at `.claude/composition/feedback/`, (iv) a structured warning containing "tautology_filter_applied" or "precondition" appears in prose output, (v) hop suggestion text MUST NOT appear in prose output â€” routing classification is in an unknown state. This case is enumerated independently from the partial correction failure post-abort test; do not rely on cross-references between the two abort paths.

The PR description MUST confirm this walk-through was completed. This protocol is a P0 prerequisite for merging materiality implementation.

## Session 29: Error Recovery Tests

### P0 Tests (must pass 2/2)

#### P0-1: Budget-precedence override (budget=1)

**Goal:** Verify the agent concludes immediately when budget is 1, regardless of server action.

**Configuration:** Evaluative posture, budget 1, any topic.

**Procedure:**
1. Invoke codex-dialogue with budget 1
2. Agent should: start conversation → receive Codex response → extract semantic data → call `process_turn` → regardless of `action` returned, proceed directly to Phase 3 synthesis
3. No follow-ups should be sent (no `codex-reply` calls after the initial response)

**Verification items:**

| # | Item | What to observe |
|---|------|-----------------|
| P0-1.1 | Agent calls `process_turn` once | Exactly one `process_turn` call |
| P0-1.2 | Agent does not call `codex-reply` | No follow-up messages sent |
| P0-1.3 | Agent proceeds to synthesis after single turn | Synthesis output produced from single-turn data |
| P0-1.4 | If `process_turn` errors, agent skips retry and synthesizes | Budget gate (DD-D5-2) prevents error recovery at budget |

#### P0-2: codex-reply failure recovery

**Goal:** Verify the agent proceeds to synthesis when `codex-reply` fails mid-conversation.

**Configuration:** Evaluative posture, budget 4, any topic.

**Procedure:**
1. Invoke codex-dialogue with budget 4
2. If `codex-reply` fails naturally during the conversation, verify recovery behavior
3. If `codex-reply` does not fail naturally: this test may not be triggerable without Codex MCP issues. Record "not exercised — codex-reply did not fail" and move to P1 tests.

**Verification items (if exercised):**

| # | Item | What to observe |
|---|------|-----------------|
| P0-2.1 | Agent does not retry `codex-reply` | No second attempt at the same follow-up |
| P0-2.2 | Agent proceeds to Phase 3 synthesis | Uses `turn_history` from completed turns |
| P0-2.3 | Agent does not call `process_turn` after failure | No new Codex response to extract from |
| P0-2.4 | Agent does not switch to `manual_legacy` mode | Transport failure after initial success stays in server-assisted mode |

**Note:** If this test is not exercisable, it becomes "spec-verified, not live-tested." The instruction review in Session 27 covers the behavior specification.

#### P0-3: manual_legacy fallback (context injection unavailable)

**Moved to Session 30** — this is a fallback mode test, not an error recovery test. Listed here for P0 tracking: it must pass 2/2 in Session 30.

### P1 Tests (must pass 1/1, one rerun allowed)

#### P1-1: checkpoint_missing via server restart

**Goal:** Verify agent handles `checkpoint_missing` when server has no in-memory state.

**Configuration:** Evaluative posture, budget 3, any topic.

**Procedure:**
1. Invoke codex-dialogue with budget 3
2. After Turn 1 succeeds, note the `state_checkpoint` and `checkpoint_id` in the response
3. If the context injection MCP server can be restarted between turns (the server process restarts, losing in-memory `AppContext`), the Turn 2 `process_turn` call with the stale checkpoint should return `checkpoint_missing` or `checkpoint_stale`
4. Verify the agent's retry behavior

**Verification items:**

| # | Item | What to observe |
|---|------|-----------------|
| P1-1.1 | Agent retries with `state_checkpoint=null` and `checkpoint_id=null` | Retry call has null checkpoint fields |
| P1-1.2 | If retry succeeds, conversation continues | Agent processes the successful response normally |
| P1-1.3 | If retry fails, agent synthesizes from `turn_history` | Proceeds to Phase 3 with Turn 1 data |

**Trigger difficulty:** Medium. Requires restarting the MCP server process between tool calls. May need manual intervention or a script. If not triggerable, record "not exercised" and rely on instruction review.

#### P1-2: Clarifier when scout_available=false

**Goal:** Verify clarifier handling works even when evidence budget is exhausted.

**Configuration:** Evaluative posture, budget 5+ (to exhaust evidence budget), topic with many entity references.

**Procedure:**
1. Run a conversation long enough for `budget.scout_available` to become false
2. If a subsequent turn's `template_candidates` contains a candidate with `scout_options: []` (clarifier), verify the agent uses the clarifier question in follow-up composition instead of attempting a scout

**Verification items:**

| # | Item | What to observe |
|---|------|-----------------|
| P1-2.1 | Agent checks for clarifier before budget gate | Clarifier check runs even when `scout_available=false` |
| P1-2.2 | Clarifier question appears in follow-up | The clarifier's text is incorporated into the follow-up message |
| P1-2.3 | No `execute_scout` call for clarifiers | Clarifiers have empty `scout_options` — no scout execution |

**Trigger difficulty:** High — depends on the conversation naturally producing a clarifier after budget exhaustion. If not exercised, record "not exercised" and rely on instruction review.

#### P1-3: ledger_hard_reject trigger attempt

**Goal:** Attempt to trigger `ledger_hard_reject` by creating conditions where the agent's extraction might mismatch.

**Configuration:** Adversarial posture, budget 4, topic where Codex makes many rapid claim changes.

**Procedure:**
1. Run a conversation with adversarial posture to maximize claim volatility
2. Observe if the agent ever produces a `process_turn` call where `focus.claims` differs from top-level `claims` (triggering the dual-claims guard)
3. If triggered, verify the agent re-examines extraction and retries once

**Verification items (if exercised):**

| # | Item | What to observe |
|---|------|-----------------|
| P1-3.1 | Agent detects `ledger_hard_reject` error | Error code recognized |
| P1-3.2 | Agent re-examines Codex response | Agent produces a corrected extraction |
| P1-3.3 | Agent retries `process_turn` with corrected extraction | Second call with different claims |
| P1-3.4 | Maximum one retry | No third attempt if retry also fails |

**Trigger difficulty:** Very high — the agent is actively trying to sync claims correctly. If not triggered after 2 conversation attempts, record "not exercised — agent correctly synced claims in all attempts" and mark as spec-verified.

#### P1-4: turn_history used for synthesis after error

**Goal:** Verify that when `process_turn` fails on a later turn, synthesis uses accumulated `turn_history` from earlier successful turns.

**Observable during:** P1-1 (checkpoint_missing). If the retry fails and the agent synthesizes from `turn_history`, verify the synthesis references Turn 1 data (claims, trajectory, position from the first successful turn).

| # | Item | What to observe |
|---|------|-----------------|
| P1-4.1 | Synthesis references data from earlier turns | Turn 1 claims and position appear in synthesis even though later turns failed |
| P1-4.2 | Synthesis acknowledges incomplete conversation | Output notes that the conversation ended early due to error |

## Session 30: Fallback Tests

### P0-3: manual_legacy fallback (2 runs required)

**Goal:** Verify the agent switches to manual_legacy mode when context injection MCP tools are unavailable.

**Configuration:** Evaluative posture, budget 3, any topic. Context injection MCP server must be stopped/unavailable.

**Procedure (run twice — 2/2 required):**

Run 1:
1. Ensure context injection MCP tools are unavailable (stop the MCP server or disconnect)
2. Invoke codex-dialogue
3. Agent should detect unavailability on the first `process_turn` call (error or timeout)
4. Agent should switch to `manual_legacy` mode for the remainder of the conversation
5. Conversation should proceed using the 3-step manual loop
6. Verify all items in the table below

Run 2:
7. Repeat steps 1-6 with the same configuration to confirm repeatability

**Verification items:**

| # | Item | What to observe |
|---|------|-----------------|
| F-1.1 | Agent attempts `process_turn` on first turn | Initial attempt to use server-assisted mode |
| F-1.2 | Agent detects unavailability | Error or timeout from `process_turn` recognized |
| F-1.3 | Agent switches to manual_legacy mode | Subsequent turns use the 3-step loop (extract → evaluate → compose), not the 7-step loop |
| F-1.4 | No further `process_turn` calls | Mode switch is permanent for this conversation |
| F-1.5 | No `execute_scout` calls | Context injection tools fully bypassed |
| F-1.6 | Synthesis produces valid output | Output format correct without evidence data or `ledger_summary` |
| F-1.7 | Conversation proceeds normally | Agent manages its own ledger, convergence detection, and turn budget |

### Mode Stability Test

**Goal:** Verify mode is not switched mid-conversation after a successful `process_turn`.

**Observable during:** Session 28 happy-path tests. If `process_turn` succeeds on Turn 1, the agent should not switch to `manual_legacy` even if later turns encounter tool errors.

| # | Item | What to observe |
|---|------|-----------------|
| F-2.1 | After successful `process_turn`, no fallback to `manual_legacy` | Even on later errors, agent stays in server-assisted mode and uses `turn_history` for synthesis |

## Spec-Verified Paths (Instruction Review Only)

The following error paths are verified by instruction review in Session 27, not live testing. The 12-point checklist in the session-split addendum covers these.

| Path | Why not live-tested | Review focus |
|------|---------------------|--------------|
| `checkpoint_stale` | Requires corrupted checkpoint data | Retry behavior: null checkpoint, max 1 retry |
| `checkpoint_invalid` | Requires malformed checkpoint string | No-retry behavior: immediate synthesis |
| `turn_cap_exceeded` | Requires 15+ turns ($30-75 in API calls) | No-retry behavior: immediate synthesis |
| Unknown action from server | Requires server modification | Treat as `conclude`, log warning |
| Cross-category retry collision | Near-theoretical under single-agent model | Per-category budgets (DD-D5-1) |
| Budget gate + error (DD-D5-2) | Would need budget exhaustion + server error simultaneously | Budget checked before error table |

**Escalation trigger:** If any spec-verified path causes issues in real usage, it moves to "must live-test" in a future session. A fault-injection harness is a P1 follow-up item (not D5 scope).
